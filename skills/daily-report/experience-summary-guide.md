# 每日经验总结写法指南

## 核心原则（必读）

**日报写给领导，经验写给自己——这是两类完全不同的文档，不可混写、不可互相替代。**

| | 日报 | 每日经验总结 |
|---|------|----------------|
| **读者** | 领导 | **自己**（未来的你、接手排查的人） |
| **目的** | 汇报进展与判断，便于决策 | **留存可复查、可复用的真实认知** |
| **详细程度** | 适度，领导能读懂即可 | **必须足够详细**，不怕长 |
| **数据** | 关键数字点缀 | **完整数据支撑**：多轮表、时间戳、日志出处 |
| **经验** | 点到为止 | **可靠的经验总结**：什么有效、什么无效、为什么 |
| **结论** | 工程判断一句话带过 | **足够可信的结论**：证据链闭合、标注置信度、写明误判 |

经验总结的价值在于：**三个月后你还能凭这份文档还原当天发生了什么、为什么那样判断、哪些结论后来被验证或推翻。** 如果写得太简略、缺数据、结论无出处，就等于没写。

**宁可多写一页有出处的细节，也不要用一句「已修复」糊弄未来的自己。**

## 与日报的区别（展开）

| 维度 | 日报 | 每日经验总结 |
|------|------|----------------|
| 读者 | 领导 | 自己 |
| 文风 | 白话、因果叙述 | 结构化、可检索、可逐条核对 |
| 数据 | 关键数字即可 | 表格 + 多轮对比 + 时间线 + 指标出处 |
| 代码 | 不写 | 文件路径、改前改后、参数值 |
| 结论 | 分散在「今日总结」 | 独立「完整结论」节，每条附数据或日志依据 |
| 失败与误判 | 写在「存在问题」 | **正文必须写清**：怎么误判的、靠什么纠正的 |
| 长度 | 一页以内为佳 | 按证据量定，通常远长于日报 |

## 推荐结构（长文模板）

```text
# YYYY-MM-DD 经验总结：<主题>

> 材料来源、主机、修订记录

## 0. 一句话结论（置信度：高/中/低）

## 1. 背景与现象（表：任务、场地、现象、影响）

## 2. 全天/多轮数据总表
   列：轮次、时长、核心指标、是否达标、备注

## 3. 推理链（按事件拆，O→H→V→C）
   O=观察 H=假设 V=验证 C=结论

## 4. 修复清单（表：层级、对象、改前、改后、验证状态）

## 5. 独立 bug / 旁路问题

## 6. 可复用排查模板（编号步骤）

## 7. 完整结论（编号，每条一句判断 + 数据支撑）

## 8. 后续行动（按优先级）

## 9. 材料索引（路径、对话存档链接）
```

另建短索引 `YYYYMMDD.md`（5–15 行）：主题、核心数据表、结论摘要、指向长文链接。

## 写作原则（写给自己的四条硬标准）

### 1. 足够详细
- 每个重大判断都有**时间戳 + 日志/快照路径**可回溯。
- 实机写**每一轮**对比，不只写最好或最差一轮。
- 排查过程写**怀疑过什么、为什么否定**，不只写最终答案。
- 代码改动写**改前改后 + 是否已实机验证**。

### 2. 完整的数据支撑
- 至少一张**全天/多轮总表**。
- 数字注明来源；数据矛盾要写出来并解释。
- 没有数据就写「待补」——**禁止无依据的定性结论**。

### 3. 可靠的经验总结
- 区分「已验证」「待验证」「已推翻」。
- 写**可复用方法**与**无效尝试/误判教训**。
- 区分**首因**与**放大因素**。

### 4. 足够可信的结论
- 独立「完整结论」节：判断 + 依据 + 置信度。
- 证据链 O→H→V→C；链未闭合则标中/低置信度。
- **主动写误判**及纠正依据。

## 取材清单（与日报共用）

1. 本机对话存档 `agent-transcripts/*.jsonl`
2. 本机 terminals（含 ssh 输出）
3. 远程 `git log` / `git diff`
4. 远程测试/日志目录（按项目约定，如会话 output、monitor 日志）
5. 用户补充的目视观察、截图说明

## 保存路径

| 文件 | 远端暂存 | 本机权威 |
|------|----------|----------|
| 长文经验总结 | `<项目根>/.cursor/工作存档/知识库/每日经验/YYYY-MM-DD-<主题>经验总结.md` | `D:\Documents\知识库\每日经验\` |
| 日索引 | `.../知识库/每日经验/YYYYMMDD.md` | 同上 |
| 日报 | `.../工作汇报/日报/YYYY-MM-DD-<主题>日报.md` | `D:\Documents\工作汇报\日报\` |

Remote-SSH 写完远端后，在本机执行 `pull-reports-to-local.ps1`（见 `work-reporting-pipeline`）。

## 交付前自检（写给自己的质量门禁）

**自问：如果明天失忆，只看这份文档，能否独立复现今天的判断？不能则继续补。**

- [ ] **详细**：时间线、怀疑→否定、改前改后
- [ ] **数据**：多轮总表、数字出处、矛盾已解释
- [ ] **经验**：可复用步骤、误判已记录
- [ ] **结论**：编号结论节，每条有依据 + 置信度
- [ ] 至少一条完整 O→H→V→C 推理链
- [ ] 短索引仅作入口，**细节在长文**
- [ ] 篇幅明显长于日报
